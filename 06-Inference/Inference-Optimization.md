# 6. 推理优化技术 (Inference Optimization)

## 6.1 推理优化八股

### 6.1.1 LLM推理的两个阶段？各自特点和瓶颈？

**参考答案：**

```
输入Prompt → [Prefill阶段] → 第一个token → [Decode阶段] → 完整输出
```

**Prefill阶段（预填充）：**
- **特点**：并行处理整个prompt，计算KV Cache
- **瓶颈**：**计算密集型**，首token时间（TTFT）
- **复杂度**：O(n²)

**Decode阶段（解码）：**
- **特点**：自回归逐个生成，复用KV Cache
- **瓶颈**：**内存带宽密集型**，每次加载全部权重
- **复杂度**：O(n) per token

**关键区别：**

| 指标 | Prefill | Decode |
|-----|---------|--------|
| 计算模式 | 并行 | 串行 |
| 瓶颈 | 计算(FLOPs) | 内存带宽 |
| GPU利用率 | 高(70-90%) | 低(10-30%) |
| 优化重点 | FlashAttention | 量化、KV Cache优化 |

---

### 6.1.2 什么是KV Cache？为什么重要？

**参考答案：**

**原理：** 缓存之前token的Key和Value，避免重复计算

```
无KV Cache：生成N个token = O(N³) 计算
有KV Cache：生成N个token = O(N) 计算
```

**显存占用计算：**
```
KV Cache = 2 × 层数 × 头数 × 头维度 × 序列长度 × batch_size × 精度

例：LLaMA-7B, seq_len=2048, batch=1, FP16
≈ 1GB
```

**重要性：**
- 加速比：**10-50倍**
- 实测：无KV Cache ~500ms/token → 有KV Cache ~15ms/token

---

### 6.1.3 PagedAttention的原理和优势？

**参考答案：**

**核心思想：** 借鉴操作系统虚拟内存，**分页管理KV Cache**

**解决的问题：**
1. **预分配浪费**：为max_seq_len预分配，实际可能只用10%
2. **内存碎片**：释放的空间无法被有效复用
3. **无法共享**：相同prompt前缀无法共享

**实现：**
- 将KV Cache分成固定大小的**block**
- 逻辑连续，物理可以不连续
- **Copy-on-Write**机制支持共享

**优势：**

| 指标 | 传统KV Cache | PagedAttention |
|-----|-------------|---------------|
| 内存利用率 | 60-70% | **95%+** |
| 并发请求数 | 10 | **30+** |
| 吞吐量 | 1000 t/s | **3000+ t/s** |

---

### 6.1.4 投机解码（Speculative Decoding）的原理？

**参考答案：**

**核心思想：** 用小模型快速生成候选token，大模型并行验证

```
草稿模型（快）→ 生成K个候选token → 目标模型（慢）→ 并行验证
```

**加速原理：**
- 草稿模型速度快5-10倍
- 目标模型一次验证多个token
- 平均接受率60-80%

**典型配置：**
- 草稿模型：1B（目标7B）或相同模型早停
- 候选数K：4-5
- 加速比：约**2倍**

**变体：**
- **Medusa**：多头预测，不依赖草稿模型
- **Lookahead Decoding**：n-gram池匹配

### 6.1.5 模型量化方法对比？

**参考答案：**

| 方法 | 精度 | 特点 | 适用场景 |
|-----|------|------|---------|
| **GPTQ** | INT4/INT3 | 逐层量化，Hessian矩阵 | GPU部署 |
| **AWQ** | INT4 | 保护显著权重，对激活感知 | **推荐**，指令模型 |
| **GGUF** | Q4/Q5/Q8 | llama.cpp格式 | **CPU/本地部署** |
| **SmoothQuant** | INT8 | 平滑激活异常值 | 服务端 |
| **BitsAndBytes** | NF4 | 动态量化 | QLoRA训练 |

**GPTQ vs AWQ：**

| 特性 | GPTQ | AWQ |
|-----|------|-----|
| 校准数据 | 需要较多(128+) | 需要较少(16-32) |
| 推理速度 | 快 | **更快** |
| 指令模型 | 可能下降 | **保持较好** |
| 推荐度 | 高 | **更高** |

**GGUF量化类型：**

| 类型 | 大小(7B) | 质量 |
|-----|---------|------|
| Q4_K_M | 4.0GB | 好 |
| Q5_K_M | 4.7GB | 很好 |
| Q8_0 | 6.7GB | 接近无损 |

### 6.1.6 连续批处理 vs 动态批处理？

**参考答案：**

**动态批处理：**
- 服务器层面组合请求
- 仍需等待整个batch完成
- 提升：2-3倍

**连续批处理（In-flight Batching）：**
- **Step级别**动态调整
- 请求随时完成随时退出
- 新的请求随时加入
- 提升：**5-10倍**

**核心区别：** 连续批处理在decode的每个step都可以调整batch。

## 6.2 解码优化

### 6.2.1 解码策略对比？

**参考答案：**

| 策略 | 特点 | 适用场景 |
|------|------|---------|
| **Greedy** | 选概率最高，确定性 | 代码生成、事实问答 |
| **Beam Search** | 维护多个候选 | 机器翻译、摘要 |
| **Sampling** | 随机采样，多样性 | 创意写作、对话 |

**采样参数：**

| 参数 | 作用 | 典型值 |
|------|------|-------|
| Temperature | 控制随机性 | 0.7-1.0 |
| Top-K | 只从top K采样 | 20-50 |
| Top-P | 从累积概率P采样 | 0.9-0.95 |

**推荐配置：**
- 代码/事实：Greedy 或 Beam Search
- 对话：Sampling (T=0.7, top_p=0.9)
- 创意：Sampling (T=0.8, top_p=0.95)

### 6.2.2 FlashAttention的原理？

**参考答案：**

**核心思想：** **IO-aware**分块计算，减少HBM和SRAM之间的数据传输

**标准Attention的问题：**
- 需要存储中间S和P矩阵
- HBM读写量O(N²)
- 内存带宽成为瓶颈

**FlashAttention优化：**
1. 将Q,K,V分块加载到SRAM
2. 在SRAM中完成注意力计算
3. 只输出最终结果到HBM
4. 使用**在线Softmax**避免存储完整矩阵

**效果：**
- 显存：O(N²) → **O(N)**
- 速度：序列越长加速越明显（4K序列约**7-8倍**）

## 6.3 模型压缩

### 6.3.1 知识蒸馏的原理？

**参考答案：**

**核心思想：** 让小模型（Student）学习大模型（Teacher）的输出分布

**软标签 vs 硬标签：**
- 硬标签：[0, 1, 0]（one-hot）
- 软标签：[0.1, 0.7, 0.2]（概率分布）
- 软标签包含更多"暗知识"

**蒸馏类型：**
1. **软标签蒸馏**：KL散度损失
2. **中间层蒸馏**：模仿隐藏状态
3. **注意力蒸馏**：模仿注意力矩阵

**Temperature作用：**
- T>1使分布更平滑
- 保留类别间相对关系
- 典型值：T=2-5

---

### 6.3.2 结构化剪枝 vs 非结构化剪枝？

**参考答案：**

| 特性 | 非结构化 | 结构化 |
|------|---------|--------|
| 粒度 | 单个权重 | 神经元/头/层 |
| 压缩比 | 高(90%+) | 中等(30-50%) |
| 硬件支持 | 需要稀疏库 | 直接支持 |
| 实际加速 | 有限 | 显著 |

**结构化剪枝类型：**
- **神经元剪枝**：移除整行/列
- **注意力头剪枝**：减少头数（如LLaMA-2的GQA）
- **层剪枝**：移除整个Transformer层

### 6.4.3 推理服务的关键性能指标？

**参考答案：**

**延迟指标：**
- **TTFT** (Time To First Token)：首token时间
- **TPOT** (Time Per Output Token)：每个输出token时间
- **总延迟** = TTFT + (N-1) × TPOT

**吞吐量指标：**
- **Token吞吐量**：tokens/second
- **请求吞吐量**：requests/second

**资源指标：**
- GPU利用率
- 显存占用
- 内存带宽利用率

**优化检查清单：**
- [ ] 连续批处理
- [ ] PagedAttention
- [ ] 量化（AWQ/GPTQ）

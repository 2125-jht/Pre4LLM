# 东方电气智能培训助手 —— RAG系统项目（面试整理版）

> **项目背景**：东方电气集团为方便新员工快速了解企业知识、行业规范和技术文档，打造基于行业知识的大模型智能问答系统。

## 一、项目概述

### 1.1 项目背景与目标

**背景**：
- 东方电气作为大型装备制造企业，拥有海量技术文档、产品手册、行业标准、培训资料

**目标**：
- 构建一套**企业知识智能问答系统**，支持自然语言提问
- 系统自动从几百份PDF文档中找到相关段落，生成回答并标注来源
- 降低新员工培训周期，提升知识获取效率

### 1.2 我的职责

- 负责RAG系统整体架构设计与核心模块开发
- 主导PDF文档解析、向量索引构建、多路召回策略实现
- 设计评测体系，持续优化系统效果

## 二、技术架构

### 2.1 系统架构图

```
用户Query → Query理解 → 多路召回 → Rerank重排 → 上下文压缩 → 生成回答 → 引用溯源
                ↓           ↓            ↓              ↓           ↓
            Query改写   向量检索    Cross-Encoder   信息去重    拒答机制
            意图识别    BM25检索                    关键句提取   引用校验
            多跳分解    混合召回                    冗余过滤
```

### 2.2 核心技术栈

| 组件 | 选型 | 备选 |
|------|------|------|
| **向量检索** | FAISS | Milvus（数据量大时） |

| **PDF解析** | PyMuPDF + pdfplumber（表格） | marker / RAGFlow DeepDoc |
| **快速原型** | LlamaIndex（仅用于搭baseline） | LangChain |

## 三、项目实现（分四周）

### 第1周：数据处理 + 基线搭建

#### 3.1.1 数据准备（企业文档场景）

**数据来源**：
- 东方电气内部技术文档、产品手册、培训资料PDF
- 能源装备行业标准、政策文件
- 总量控制在 **200-300份PDF**，覆盖风电、水电、核电、气电等主要业务板块

**PDF解析难点**：
- 技术文档多为多栏排版，纯文本split会导致内容混乱
- 表格数据密集，需要保留结构信息
- 页眉页脚需要过滤

**解决方案**：
- 使用 **PyMuPDF** 提取文本，**pdfplumber** 解析表格
- 参考 RAGFlow 的 DeepDoc 模块，学习其版面分析思路
- 实现"表格保护"切块策略，确保表格行不被切断

**实现思路**：
1. 用PyMuPDF提取文本块，保留位置信息（bbox坐标）
2. 根据x坐标判断栏数，按阅读顺序排序（先左栏后右栏）
3. 用pdfplumber提取表格，转换为markdown格式
4. 过滤页眉页脚（正则匹配"第X页"、"Page X"等）

#### 3.1.2 切块实验（面试高频考点）

**实验设计**：
- 对比3组 chunk_size：**256 / 512 / 1024**
- 对比 overlap：**50 / 100 / 200**
- 记录每组切块数量、平均长度、以及后续召回的 Recall@5 变化

**企业文档特殊点**：
- 技术参数表格不能被切断，需要"表格保护"切块策略
- 输出：一张切块策略对比表

**智能切块实现思路**：
1. 识别表格边界（正则匹配markdown表格格式）
2. 表格作为一个整体chunk，不切分
3. 普通文本按chunk_size切分，在语义边界处切分（找句号、换行）
4. 相邻chunk保留overlap，保证上下文连续性

**切块策略对比结果**：

| chunk_size | overlap | chunk数量 | 平均长度 | Recall@5 | 备注 |
|------------|---------|-----------|----------|----------|------|
| 256 | 50 | 8,520 | 198 | 72% | 切得太碎，语义不完整 |
| **512** | **100** | **4,680** | **456** | **87%** | **最优选择** |
| 1024 | 200 | 2,340 | 892 | 81% | 语义完整但召回精度下降 |

**面试话术**：
> "我做了三组对比实验，最终选择512+100的组合。256虽然chunk数量多，但切得太碎导致语义不完整；1024虽然语义完整，但召回时容易引入噪声。512在Recall@5上表现最优，达到87%。"

#### 3.1.3 层次化分块（Hierarchical Chunking）

**问题背景**：
传统固定长度切块存在明显缺陷：
- 可能切断语义相关的段落（如"技术优势"和"应用案例"被分到不同chunk）
- 缺乏文档结构信息（标题层级、章节关系）
- 无法处理跨chunk的指代消解（如"该技术"需要前文）

**解决方案**：层次化分块策略

层次化分块的核心思想是：**保持文档的层级结构，构建父子关系索引**。具体实现分为三个层次：

1. **文档级（Document）**：整篇PDF文档
2. **章节级（Section）**：按标题层级切分（H1、H2、H3）
3. **段落级（Paragraph）**：章节内的细粒度chunk

**实现方案**：

```
层次化结构示例：
Document: "H100型海上风机技术规范.pdf"
├── Section 1: "产品概述" (H1)
│   ├── Chunk 1.1: "H100型风机是东方电气..."
│   └── Chunk 1.2: "主要技术参数如下..."
├── Section 2: "技术规格" (H1)
│   ├── Section 2.1: "机械参数" (H2)
│   │   ├── Chunk 2.1.1: "叶轮直径120米..."
│   │   └── Chunk 2.1.2: "塔筒高度120.5米..."
│   └── Section 2.2: "电气参数" (H2)
│       └── Chunk 2.2.1: "额定功率10MW..."
└── Section 3: "运维要求" (H1)
    └── Chunk 3.1: "定期维护周期为..."
```

**技术实现细节**：

1. **标题识别**：用正则匹配章节编号（如"第1章"、"1.1"、"1.1.1"）
2. **层级构建**：维护一个栈结构，遇到新标题时弹出同级或更低级的章节
3. **chunk创建**：标题单独作为一个chunk（用于粗粒度召回），内容细粒度切分
4. **父子关系**：每个chunk记录parent_id，支持上下文获取

**层次化分块的优势**：

| 特性 | 传统切块 | 层次化分块 |
|------|----------|------------|
| 结构信息 | 无 | 保留章节层级 |
| 召回粒度 | 单一 | 粗粒度+细粒度 |
| 上下文获取 | 固定overlap | 动态上下文窗口 |
| 指代消解 | 困难 | 通过父chunk解决 |
| 索引大小 | N | 约1.3N（标题chunk额外） |

**面试话术**：
> "我在项目中实现了层次化分块，核心思想是保留文档的层级结构。比如技术文档有'第1章 产品概述'、'1.1 技术参数'这样的层级，我会把标题单独作为一个chunk（用于粗粒度召回），内容细粒度切分（用于精确定位）。这样当用户问'该技术的优势'时，系统可以通过父chunk找到'该技术'指代的是什么。"

#### 3.1.4 语义边界检测（Semantic Boundary Detection）

**问题背景**：
固定长度切块的最大问题是**切断语义边界**——一个完整的语义单元被强行分割到两个chunk中，导致：
- 召回时只能拿到半个语义单元
- 生成时上下文不完整，容易产生幻觉

**解决方案**：基于语义相似度的动态切分

核心思想：**相邻句子如果语义相似度高，应该属于同一个chunk；语义相似度突变，说明是边界**。

**技术实现**：

1. **句子切分**：按中英文句子结束符切分（。！？；.!?）
2. **Embedding计算**：用bge-small-zh-v1.5计算每句的embedding
3. **相似度计算**：计算相邻句子的余弦相似度
4. **边界检测**：满足以下条件之一视为边界：
   - 相似度低于阈值（如0.7）
   - 局部最小值（比前后都低）
   - 相似度骤降超过50%
5. **后处理**：合并短chunk（<200字）、拆分长chunk（>800字）

**语义边界检测效果对比**：

| 切分方式 | 语义完整度 | Recall@5 | 平均chunk长度 | 边界准确率 |
|----------|------------|----------|---------------|------------|
| 固定512 | 72% | 78% | 456 | - |
| 语义边界 | **91%** | **85%** | 523 | **87%** |

**边界准确率定义**：人工标注100个真实边界，检测正确的比例。

**面试话术**：
> "语义边界检测是我项目中的一个亮点。传统固定长度切块会切断语义，比如'该技术的优势包括：高效率、低成本。'可能被切成两半。我的方案是：先按句子切分，用embedding计算相邻句子相似度，相似度突变的地方就是边界。这样切出来的chunk语义更完整，边界准确率达到87%。"

#### 3.1.5 向量索引构建

- 使用 **FlagEmbedding** 的 bge-large-zh-v1.5 做 embedding
- 用 **FAISS** 构建索引（先用 Flat，后面对比 IVF/HNSW）
- 输出：索引构建脚本 + 向量维度和文档数记录

**实现思路**：

1. **Embedding生成**：
   - 使用bge-large-zh-v1.5（1024维）
   - query添加前缀："为这个句子生成表示以用于检索相关文章："
   - batch_size=32，使用FP16加速
   - 归一化embedding（用于余弦相似度）

2. **FAISS索引选择**：
   - **Flat**：暴力搜索，精度最高，适合<10万数据
   - **IVF**：倒排文件索引，平衡精度和速度
   - **HNSW**：图索引，查询最快，适合>100万数据

3. **索引构建流程**：
   - 生成所有chunk的embedding
   - 根据数据量选择索引类型
   - 添加向量到索引
   - 保存索引文件和chunk元数据

### 第2周：Query理解 + 多路召回 + 召回评测

#### 3.2.0 Query理解与改写（基于大模型的方法）

**问题背景**：
用户原始Query往往存在以下问题：
- **表述不规范**："风机叶片多长" vs "风机叶片的尺寸参数"
- **缺少上下文**：多轮对话中的"它"指代不明
- **意图模糊**："H100怎么样"（是问性能？价格？还是可靠性？）
- **多跳问题**："H100和H150哪个更适合海上"（需要分别查两款产品再对比）

**解决方案**：Query理解流水线（基于大模型）

```
原始Query → 意图识别 → 指代消解 → Query改写（LLM） → 多跳分解 → 改写后Query(s)
```

**1. 意图识别（Intent Classification）**

将用户Query分类为不同类型，采用不同召回策略：

| 意图类型 | 示例 | 召回策略 |
|----------|------|----------|
| FACTUAL | "H100的额定功率是多少" | recall_top_k=10，混合召回 |
| COMPARISON | "直驱和双馈风机的区别" | recall_top_k=20，扩展查询词 |
| SUMMARY | "海上风电的发展趋势" | recall_top_k=15，多样性召回 |
| HOW_TO | "如何更换风机齿轮箱" | recall_top_k=15，混合召回 |
| MULTI_HOP | "H100比H150轻多少" | recall_top_k=30，需要分解 |
| CLARIFICATION | "这个怎么样" | 需要澄清 |

**实现方式**：基于规则+LLM分类
- 先用关键词正则快速分类
- 不确定时调用LLM进行意图识别

**2. 指代消解（Coreference Resolution）**

解决多轮对话中的指代问题：

**实现思路**：
1. 检测Query中是否包含指代词（它、这、那、该、此）
2. 从对话历史中提取实体（产品型号、技术术语、引号内容）
3. 用LLM判断指代关系，替换指代词

**示例**：
- 历史："介绍一下H100型风机"
- Query："它的功率是多少？"
- 消解后："H100型风机的功率是多少？"

**3. Query改写（基于大模型）**

**核心思想**：使用大模型生成多个语义等价的Query变体，提升召回覆盖率。

**实现方案**：

```python
# Prompt模板
QUERY_REWRITE_PROMPT = """你是一个专业的Query改写助手。请将用户问题改写为3-5个不同表述的搜索查询，用于检索相关技术文档。

要求：
1. 保持原意不变，不要改变问题的核心含义
2. 使用不同的词汇和句式，覆盖更多检索可能
3. 适合文档检索，使用正式、规范的技术术语
4. 针对能源装备领域的专业术语进行扩展

用户问题：{query}
问题类型：{intent}

请输出改写后的查询（每行一个）：
"""
```

**改写策略**：

| 意图类型 | 改写策略 | 示例 |
|----------|----------|------|
| FACTUAL | 同义词扩展、句式变换 | "功率"→"额定功率/容量/出力" |
| COMPARISON | 对比角度扩展 | "区别"→"差异/对比/优缺点/不同点" |
| SUMMARY | 汇总角度扩展 | "发展趋势"→"发展方向/技术路线/未来展望" |
| MULTI_HOP | 分解为子查询 | "A比B轻多少"→["A的重量","B的重量"] |

**示例效果**：

| 原始Query | LLM改写结果 |
|-----------|-------------|
| "风机叶片多长" | 1. 风机叶片的尺寸参数<br>2. 风机叶片的长度规格<br>3. 风机叶轮直径是多少 |
| "H100怎么样" | 1. H100型风机的性能特点<br>2. H100型风机的技术参数<br>3. H100型风机的优缺点 |
| "海上风电基础类型" | 1. 海上风电基础形式<br>2. 海上风电基础种类<br>3. 海上风电基础分类 |

**4. Query理解流水线整合**

**处理流程**：

1. **Step 1 - 指代消解**：用LLM或规则消解指代词
2. **Step 2 - 意图识别**：分类Query类型
3. **Step 3 - Query改写**：调用LLM生成多个Query变体
4. **Step 4 - 多跳检测**：如果是多跳问题，分解为子Query

**输出结构**：
```json

    "resolved_query": "H100型风机的功率是多少？",
    "intent": "FACTUAL",
    "rewritten_queries": [
        "H100型风机的功率是多少",
        "H100型风机的额定功率",

    ],
    "search_strategy": {
        "recall_top_k": 10,
        "use_bm25": true,
        "use_vector": true,

    },
    "is_multi_hop": false,
    "sub_queries": []
}
```

**Query理解效果**：

| 场景 | 原始Query | 处理后 | 效果 |
|------|-----------|--------|------|
| 指代消解 | "它的功率是多少？" | "H100型风机的功率是多少？" | 召回率+12% |
| LLM改写 | "基础类型" | "基础类型/形式/种类/分类" | 召回率+15% |
| 多跳分解 | "H100比H150轻多少" | ["H100重量", "H150重量"] | 准确率+15% |

**面试话术**：
> "Query理解是我项目的重要模块。用户Query往往不规范，比如问'它怎么样'，系统需要知道'它'指代什么。我实现了三级处理：意图识别（区分事实型/对比型/多跳型）、指代消解（从对话历史找实体）、**Query改写（基于大模型生成多个语义等价变体）**。经过这三步，召回率平均提升10%以上。"

#### 3.2.1 实现3路召回

| 路线 | 方法 | 说明 |
|------|------|------|
| **路线A** | 纯向量召回（FAISS） | 语义匹配 |
| **路线B** | 纯 BM25 召回 | 关键词匹配 |
| **路线C** | 混合召回（向量 + BM25 分数加权融合） | 综合最优 |

**BM25召回实现思路**：

1. **中文分词**：使用jieba对文档和Query分词
2. **索引构建**：用rank_bm25的BM25Okapi构建索引
3. **分数计算**：计算Query与每篇文档的BM25分数
4. **Top-K返回**：取分数最高的K篇文档

**混合召回融合策略（RRF）**：

RRF（Reciprocal Rank Fusion）是一种简单有效的融合方法：

```
RRF_score(d) = Σ(1 / (k + rank_d))
```

其中k=60是平滑因子，rank_d是文档d在某个召回源中的排名。

**融合流程**：
1. 向量召回Top50，记录排名
2. BM25召回Top50，记录排名
3. 对每个文档计算RRF分数
4. 按RRF分数排序，取Top-K

#### 3.2.2 构建评测集

**评测集设计**：
- 自建 **80-120条** (query, ground_truth_doc_id, answer) 三元组
- Query分三类：
  - **事实型**："东方电气2023年风电装机量是多少"
  - **对比型**："直驱风机和双馈风机各有什么优缺点"
  - **汇总型**："海上风电技术2025年主要发展趋势"

> 面试技巧：强调"我的评测集覆盖三类问题，其中对比型和汇总型是纯向量召回的弱点"——自然引出为什么要做混合召回

#### 3.2.3 跑评测

**评测指标**：
- **Recall@K**：正确答案在Top-K中的比例
- **MRR**：平均倒数排名
- **NDCG**：归一化折损累计增益

**评测结果**：

| 召回方案 | Recall@3 | Recall@5 | Recall@10 | 备注 |
|----------|----------|----------|-----------|------|
| 纯向量 | 68% | 72% | 78% | baseline |
| 纯BM25 | 65% | 70% | 76% | 关键词场景更强 |
| 混合召回 | **78%** | **87%** | **92%** | 通常最优 |

**按问题类型分析**：

| 问题类型 | 纯向量Recall@5 | 纯BM25 Recall@5 | 混合Recall@5 |
|----------|----------------|-----------------|--------------|

#### 3.3.1 接入Reranker

- 使用 **FlagEmbedding** 的 bge-reranker-v2-m3

1. **Cross-Encoder架构**：Query和Doc一起输入，输出相关性分数
2. **重排流程**：
   - 先用向量/BM25召回Top20候选

   - 按分数重排，取Top5

| 方案 | Recall@5 | 准确率 | 说明 |
|------|----------|--------|------|
| 直接Top5 | 87% | 72% | 无重排 |

#### 3.3.2 上下文压缩（Context Compression）

**问题背景**：
召回的Top-K文档往往存在以下问题：
- **内容冗余**：多个文档包含重复信息
- **信息密度低**：长段落中只有几句话与Query相关
- **超出上下文窗口**：5个chunk × 512字 = 2560字，接近7B模型的4K上限

**解决方案**：上下文压缩流水线

```
召回文档 → 信息去重 → 相关性过滤 → 关键句提取 → 压缩后上下文
```

**1. 信息去重（Deduplication）**

**实现思路**：
1. 计算文档两两之间的embedding相似度
2. 相似度>0.85视为重复
3. 保留Reranker分数最高的，去除其他

**2. 相关性过滤（Relevance Filtering）**

**实现思路**：
1. 计算Query与每篇文档的embedding相似度
2. 低于阈值（如0.5）直接丢弃
3. 返回高相关性文档

**3. 关键句提取（Key Sentence Extraction）**

**实现思路**：
1. 按句子切分文档
2. 计算每句与Query的embedding相似度
3. 取Top-3句，保持原文顺序

**上下文压缩效果**：

| 指标 | 压缩前 | 压缩后 | 效果 |
|------|--------|--------|------|
| 平均文档数 | 5 | 4.2 | -16% |
| 平均token数 | 2560 | 980 | **-62%** |
| 回答准确率 | 82% | **88%** | **+6pp** |
| 响应延迟 | 2.5s | **1.8s** | **-28%** |

**准确率提升原因**：
- 去除冗余信息，减少噪声干扰
- 保留高相关性内容，提升信噪比
- 缩短上下文，让模型更聚焦

**面试话术**：
> "上下文压缩是我项目的一个重要优化。召回的5个chunk可能有重复内容，而且每个chunk只有几句话与Query真正相关。我的方案是三级压缩：先去重（相似度>0.85视为重复），再过滤低相关性文档，最后提取关键句。压缩后token数减少62%，但回答准确率反而从82%提升到88%，因为去除了噪声。"

#### 3.3.3 生成约束

**实现功能**：
1. **引用溯源**：回答中标注证据来源段落
2. **拒答机制**：召回分数全低于阈值时返回"我不确定"而不是硬编

> 面试高频问题："你怎么缓解幻觉？"
> 
> 回答："证据绑定 + 低置信度拒答 + 生成后引用校验"

**Prompt设计**：

```
你是一个专业的能源装备领域技术助手。请根据提供的参考资料回答问题。

要求：
1. 只基于提供的参考资料回答，不要编造信息
2. 如果参考资料不足以回答问题，请明确说明
3. 回答时标注信息来源，格式为[文档X]

{context}

```

**拒答机制**：
- 设置置信度阈值（如0.3）
- 如果所有召回文档的分数都低于阈值，返回拒答模板
- 拒答模板："根据现有资料，我无法确定这个问题的答案。建议您查阅相关技术文档或咨询专业人员。"

#### 3.3.4 vLLM + 模型量化本地部署（Qwen2.5-14B）

**问题背景**：
生产环境部署大模型面临以下挑战：
- **显存占用高**：Qwen2.5-14B FP16需要28GB+显存
- **推理速度慢**：单卡QPS低，无法满足并发需求
- **部署成本高**：需要多张A100，成本不可接受

**解决方案**：vLLM + AWQ/GPTQ量化

**1. vLLM推理引擎**

vLLM是伯克利大学开源的高吞吐大模型推理引擎，核心特性：
- **PagedAttention**：借鉴操作系统虚拟内存管理，减少显存碎片
- **Continuous Batching**：动态批处理，提升吞吐量
- **Speculative Decoding**：投机采样，加速解码

**2. 模型量化（AWQ/GPTQ）**

量化是将模型权重从高精度（FP16）转换为低精度（INT4/INT8），减少显存占用和提升推理速度。

| 量化方法 | 精度 | 显存占用 | 速度提升 | 质量损失 |
|----------|------|----------|----------|----------|
| FP16 | 16bit | 28GB | 1x | 0% |
| INT8 | 8bit | 14GB | 1.5x | <1% |
| GPTQ 4bit | 4bit | 8GB | 2.5x | 2-3% |
| AWQ 4bit | 4bit | 8GB | 2.5x | 1-2% |

**AWQ量化原理**：
- 识别并保护重要的权重通道（激活值大的通道）
- 对非重要通道进行更激进的量化
- 相比GPTQ，AWQ对重要权重保护更好，质量损失更小

**RAG系统集成**：

```python
class vLLMRAGGenerator:
    """基于vLLM的RAG生成器"""
    
    def __init__(self, 
                 model_path: str = "Qwen/Qwen2.5-14B-Instruct-AWQ",
                 quantization: str = "awq"):
        self.llm = LLM(
            model=model_path,
            quantization=quantization,
            tensor_parallel_size=1,
            gpu_memory_utilization=0.85,
            max_model_len=4096,
            trust_remote_code=True,
        )
        self.sampling_params = SamplingParams(
            temperature=0.3,
            top_p=0.9,
            max_tokens=512,
            repetition_penalty=1.1,
        )
    
    def generate(self, query: str, retrieved_docs: List[Dict], 
                 confidence_threshold: float = 0.3) -> Dict:
        # 检查置信度
        max_score = max(d.get("rerank_score", 0) for d in retrieved_docs) if retrieved_docs else 0
        
        if max_score < confidence_threshold:
            return {
                "answer": "根据现有资料，我无法确定这个问题的答案。",
                "confidence": max_score,
                "sources": []
            }
        
        # 构建prompt
        prompt = self._build_rag_prompt(query, retrieved_docs)
        
        # vLLM生成
        outputs = self.llm.generate([prompt], self.sampling_params)
        answer = outputs[0].outputs[0].text
        
        return {
            "answer": answer.strip(),
            "confidence": max_score,
            "sources": [...]
        }
```

## 五、面试高频问题与回答

### 5.1 技术细节类

#### Q1：chunk size 怎么选的？

**完整回答**：
> "我做了三组对比实验，chunk_size分别取256/512/1024，overlap取50/100/200。最终选择512+100的组合，因为：
> - 256切得太碎，语义不完整，Recall@5只有72%
> - 1024虽然语义完整，但召回精度下降，Recall@5只有81%
> - 512在Recall@5上表现最优，达到87%
> 
> 另外，我还实现了语义边界检测——用embedding计算相邻句子相似度，相似度突变的地方就是边界。这样切出来的chunk语义更完整。"

#### Q2：层次化分块是什么？有什么用？

**完整回答**：
> "层次化分块的核心是保留文档的层级结构。传统切块把文档切成平级的chunk，丢失了章节信息。我的方案是：
> - 文档级 → 章节级（H1/H2/H3标题） → 段落级
> - 标题单独作为一个chunk，用于粗粒度召回
> - 段落chunk保留父节点ID，用于上下文获取
> 
> 好处是：1）召回可以分粗细粒度；2）解决指代消解问题（'该技术'可以通过父chunk找到指代）；3）上下文窗口更灵活。"

#### Q3：语义边界检测怎么做？

**完整回答**：
> "语义边界检测解决的是'在哪里切分'的问题。我的实现：
> 1. 按句子切分文档
> 2. 用embedding模型计算相邻句子的相似度
> 3. 相似度低于阈值，或出现局部最小值，视为边界
> 4. 在边界处切分，保证每个chunk内部语义连贯
> 
> 这样相比固定长度切块，语义完整度从72%提升到91%。"

#### Q4：Query理解与改写有哪些策略？（重点）

**完整回答**：
> "Query理解我实现了三个模块：
> 1. **意图识别**：区分事实型/对比型/汇总型/多跳型，不同意图用不同召回策略
> 2. **指代消解**：从对话历史中提取实体，替换'它/这/那'等指代词
> 3. **Query改写（基于大模型）**：调用LLM生成3-5个语义等价的Query变体，覆盖更多检索可能
> 
> **为什么选择LLM改写而不是规则？**
> - 规则改写需要维护大量同义词词典，难以覆盖所有领域术语
> - LLM能理解语义等价关系，比如'类型'和'形式'是同义词
> - LLM能进行句式变换，比如'风机叶片多长'→'风机叶片尺寸参数'
> 
> 比如用户问'它的功率是多少'，系统会先消解为'H100的功率是多少'，再调用LLM改写为'H100额定功率'、'H100容量参数'等多个变体，召回率平均提升15%以上。"

#### Q5：上下文压缩怎么做？有什么好处？

**完整回答**：
> "上下文压缩解决的是召回文档冗余、信息密度低的问题。我的方案是三级压缩：
> 1. **去重**：相似度>0.85的文档视为重复，只保留分数最高的
> 2. **相关性过滤**：低于阈值的文档直接丢弃
> 3. **关键句提取**：从每篇文档中提取与Query最相关的3句话
> 
> 压缩后token数减少62%，但回答准确率反而从82%提升到88%，因为去除了噪声，信噪比更高。"

#### Q6：vLLM和模型量化怎么做的？（重点）

**完整回答**：
> "部署方面我采用了vLLM+AWQ量化的方案：
> - **模型选择**：Qwen2.5-14B，相比7B推理能力更强
> - **vLLM**：伯克利开源的推理引擎，PagedAttention技术减少显存碎片，Continuous Batching提升吞吐量
> - **AWQ量化**：4bit量化，把显存占用从28GB降到10GB，质量损失仅1-2%
> 
> **为什么选择14B而不是7B？**
> - 14B模型推理能力更强，能更好处理复杂Query改写和多跳推理
> - 量化后显存占用可控（10GB），单卡A100即可部署
> - 质量损失仅1-2%，性价比很高
> 
> 最终单卡A100可以达到12 QPS，比transformers原生推理提升6倍。"

#### Q7：为什么不只用向量检索？

**完整回答**：
> "举例子——用户搜'海上风电基础类型'，文档里写的是'海上风电基础形式'，纯向量可能召不回来，但BM25能匹配到'海上风电'和'基础'这两个关键词。
> 
> 从评测数据看，对比型问题（如'直驱和双馈风机的区别'）纯向量Recall@5只有58%，加上BM25后提升到82%。所以混合召回效果最稳，能互补两种召回方式的弱点。"

#### Q8：Reranker到底有多大提升？

**完整回答**：
> "Top20召回后Rerank取Top5，相比直接Top5，准确率从72%提升到86%，提升了14个百分点。
> 
> 这是因为Reranker是Cross-Encoder架构，能捕捉query和doc的细粒度交互，比双塔模型的向量相似度更精准。但代价是推理速度较慢，所以我们采用'先粗筛再精排'的策略控制延迟。"

#### Q9：空召回你怎么处理？

**完整回答**：
> "三步走——**LLM Query改写** + 同义词扩展 + 降级拒答。
> 
> 调用LLM把'基础类型'改写为'基础形式/基础种类/基础分类'等多个变体。如果改写后仍然召不回，就触发拒答机制，返回'根据现有资料无法回答'，而不是硬编一个答案。"

#### Q10：幻觉怎么缓解？

**完整回答**：
> "三板斧——证据绑定 + 低置信度拒答 + 生成后引用校验。
> 
> 1）prompt里明确要求'只基于参考资料回答'；2）设置置信度阈值，低于阈值时拒答；3）生成后校验数字是否与原文一致。这套组合拳下来，幻觉问题基本可控。"

#### Q11：QPS多少，瓶颈在哪？

**完整回答**：
> "单卡A100上QPS约12（Qwen2.5-14B），主要瓶颈在重排模型和生成模型。
> 
> 优化方案：1）先粗筛再精排，减少重排文档数；2）缓存热查询；3）vLLM+AWQ量化。如果数据量增大，可以考虑把向量库迁移到Milvus，支持分布式部署。"

### 5.2 项目设计类

#### Q12：为什么选择这个技术栈？

**完整回答**：
> - **Embedding选bge-large-zh-v1.5**：中文场景SOTA，C-MTEB榜单第一，开源免费
> - **向量库选FAISS**：轻量、易部署，适合我们200-300份文档的规模
> - **重排选bge-reranker-v2-m3**：与Embedding同系列，兼容性最好，效果也不错
> - **生成选Qwen2.5-14B**：中文能力强，14B规模推理能力优于7B，量化后部署成本低
> - **推理引擎选vLLM**：吞吐量大，PagedAttention减少显存碎片
> - **量化选AWQ**：4bit量化质量损失小，显存占用低

#### Q13：如果数据量增大10倍，你会怎么优化？

**完整回答**：
> "1）向量库从FAISS迁移到Milvus，支持分布式；2）索引从Flat升级到HNSW，降低查询延迟；3）引入缓存层，热点查询直接返回；4）重排模型做量化或蒸馏，提升吞吐量；5）考虑引入ElasticSearch做关键词召回的分布式支持。"

#### Q14：怎么保证回答的准确性？

**完整回答**：
> "四道防线：1）检索端：混合召回+重排，确保召回质量；2）上下文压缩：去重+过滤+关键句提取，提升信噪比；3）生成端：引用溯源，每个回答都标注来源；4）评测端：自建评测集，持续监控badcase；5）兜底：置信度阈值，不确定时主动拒答。"

### 5.3 场景理解类

#### Q15：企业文档和金融研报有什么区别？

**完整回答**：
> - **文档类型**：企业文档更杂，有技术规范、产品手册、培训资料
> - **表格特点**：企业文档表格多为技术参数，需要精确解析，不能错位
> - **专业术语**：能源装备行业有大量专有名词，如IGBT、变流器、齿轮箱
> - **更新频率**：企业文档相对稳定，可以离线构建索引

#### Q16：如果用户问的是文档里没有的，怎么办？

**完整回答**：
> "1）召回分数低于阈值时，返回'根据现有资料无法回答'；2）可以引导用户：'您的问题可能涉及XX领域，建议咨询XX部门'；3）记录这类问题，作为后续文档补充的参考。关键是不要硬编，宁可拒答也不要给错误信息。"

### 5.4 进阶问题

#### Q17：Embedding和BM25的本质区别是什么？

**完整回答**：
> "BM25是基于词频的稀疏检索，匹配关键词，对同义词无能为力；Embedding是基于语义向量的稠密检索，能捕捉语义相似性，但对精确关键词匹配不如BM25稳定。
> 
> 举个例子：'风机'和'风力发电机'，BM25可能匹配不到，但Embedding可以；但'2023年营收100亿'这种精确数字，BM25更可靠。所以混合召回能互补。"

#### Q18：Reranker为什么比向量相似度更准？

**完整回答**：
> "向量检索是双塔结构，query和doc分别编码，交互只在最后点积；Reranker是Cross-Encoder，query和doc一起输入，每一层都有交互，能捕捉更细粒度的匹配信号。
> 
> 代价是Cross-Encoder计算量大，不能用于大规模召回，只能用于重排阶段。"

#### Q19：你的项目还有什么可以改进的地方？

**完整回答**：
> "还有几个可以优化的方向：1）领域微调，用企业文档微调Embedding模型；2）引入GraphRAG处理复杂多跳问题；3）在线学习，根据用户反馈持续优化；4）如果数据量增大，迁移到Milvus+HNSW。"

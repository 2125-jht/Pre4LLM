# 强化学习算法全景：从策略梯度到R1复现

> 本文系统梳理从基础策略梯度到DeepSeek-R1复现的核心算法，重点深入讲解 PPO、DPO、GRPO、DAPO 在大模型RL中的应用。

---

## 第一部分：策略梯度基础

### 1.1 策略梯度核心直觉

#### 一句话理解策略梯度
**好的行为多做，坏的行为少做。**

想象你是一个学做菜的新手：
- 你随机尝试各种做法（采样）
- 朋友试吃后给打分（奖励）
- 打分高的做法，下次多用（增大概率）
- 打分低的做法，下次少用（减小概率）

这就是策略梯度的全部思想！

| 概念 | 类比 | 数学表达 |
|------|------|----------|
| 策略 | 你做菜的方法 | 概率分布 π(a\|s) |
| 梯度 | 怎么调整这个分布 | ∇θ |
| 奖励 | 朋友的打分 | R |

**核心公式：**
```
∇J(θ) = E[ A · ∇log π ]
```
- ∇log π：增大某动作概率的方向
- A（优势）：这个动作比平均水平好多少
- E：对多次采样取平均

### 1.2 SFT vs 策略梯度

**SFT的梯度（你已经很熟了）**
```
∇L_SFT = -∇log π(a*|s; θ)
```
对每个标准答案 a*，增大其概率

**策略梯度只多了一个「权重」**
```
∇J_PG = -R · ∇log π(a|s; θ)
```
- a 不是标准答案，而是模型自己采样的！
- R 是这个采样结果的奖励分数

**记忆口诀**
| 方法 | 特点 |
|------|------|
| SFT | 老师说什么就学什么（有标准答案） |
| PG | 自己试，好的多学坏的少学（试错学习） |

**区别：** SFT的权重恒为1（每条数据同等重要），策略梯度的权重是R（好数据权重大、坏数据权重小甚至为负）。

### 1.3 REINFORCE：最简策略梯度算法

只需4步，就能实现策略梯度：

| 步骤 | 操作 |
|------|------|
| Step 1 | 用当前策略 π(θ) 采样一条轨迹 |
| Step 2 | 计算这条轨迹的总奖励 R |
| Step 3 | 计算梯度 g = R · ∇log π(a\|s;θ) |
| Step 4 | 更新参数 θ ← θ + α · g |

#### 直觉解读每一步
- **∇log π(a\|s;θ)** 指明了「增大动作a概率」的方向
- **乘以 R > 0**：沿这个方向走 → 增大好动作概率
- **乘以 R < 0**：反方向走 → 减小坏动作概率
- **R = 0**：不更新（中性结果不提供信息）

#### 问题
**方差极大！** 一条轨迹的R波动很大，梯度估计不稳定。→ 需要baseline和更好的advantage估计。

### 1.4 从R到Advantage

原始策略梯度直接用奖励R，但R波动很大（方差大）。改进方法是引入**Advantage（优势）**：

```
A = R - baseline
```

**直观理解**：
- 全班平均75分（baseline），你考95分 → A = +20（远超平均，值得学习）
- 你考55分 → A = -20（低于平均，需要改进）

减去baseline不改变梯度的正确性，但大幅降低方差，训练更稳定。

## 第二部分：大模型RL算法概览

### 2.1 大模型RL的特点

与传统游戏/机器人RL不同，大模型场景有独特挑战：

| 特点 | 游戏RL | 大模型RL |
|------|--------|----------|
| 动作空间 | 离散且小（如上下左右） | 极大（词表几万个token） |
| 奖励密度 | 密集（每步可能有奖励） | **极其稀疏**（只有最后有分） |
| Episode长度 | 几百步 | 几十到几千token |
| 奖励来源 | 环境自动给出 | 需要Reward Model或规则判断 |

**稀疏奖励的影响**：
- 生成100个token，只有最后一个token有奖励（如答案正确+1，错误0）
- 所有中间token的「未来奖励」基本一样（都是最终奖励的折扣）
- 因此GAE的作用被削弱，很多实现会简化或不用

### 2.2 算法演进路线

```
REINFORCE (1992)
    ↓
Actor-Critic (2016) —— 引入Critic估计价值
    ↓
PPO (2017) —— Clip限制更新幅度，工业界主流
    ↓
    ├─→ DPO (2023) —— 跳过RL，直接用偏好数据优化
    │
    └─→ GRPO (2024) —— 去掉Critic，组内归一化，省显存
              ↓
         ├─→ DAPO (2025) —— 四项工程优化，训练更稳定
         ├─→ Dr.GRPO (2025) —— 修正优势估计偏差
         ├─→ GFPO (2025) —— 多采样少思考，拒绝冗长
         └─→ GSPO (2025) —— MoE架构的重要性采样对齐
```

| 算法 | 模型数量 | 核心特点 | 适用场景 |
|------|----------|----------|----------|
| PPO | 4个 | 成熟稳定，有Critic精细估计 | 资源充足，追求稳定 |
| DPO | 2个 | 无需在线采样，直接用偏好对 | 数据固定，快速迭代 |
| GRPO | 2个 | 去Critic，组内相对排序 | 可验证任务（数学/代码）|
| DAPO | 2个 | GRPO+四项改进 | GRPO训练不稳定时 |
| Dr.GRPO | 2个 | 移除长度/难度偏差 | 需要更准确优势估计时 |
| GFPO | 2个 | 拒绝采样+简洁性筛选 | 解决答案冗长问题 |
| GSPO | 2个 | Sequence-Level采样 | MoE架构模型 |

## 第三部分：PPO for LLM

### 3.1 四模型架构

PPO训练需要同时维护4个模型：

```
┌─────────────────────────────────────────────────────────┐
│                    PPO for LLM                          │
├─────────────────────────────────────────────────────────┤
│  Policy Model (π_θ)      ← 被训练的策略，生成回答         │
│  Reference Model (π_ref) ← 冻结的SFT模型，防止偏离太远    │
│  Critic Model (V)        ← 价值网络，估计每个token的价值  │
│  Reward Model (R)        ← 奖励模型，给整个回答打分       │
└─────────────────────────────────────────────────────────┘
```

**各模型作用**：
- **Policy**：被优化的主角，学习生成高奖励回答
- **Reference**：提供KL约束，防止Policy跑得太偏
- **Critic**：估计每个token的「未来能拿多少分」，用于计算Advantage
- **Reward Model**：给完整回答打分（如答案正确性、 helpfulness等）

**显存开销**：4个模型同时加载，是大模型RL的主要瓶颈。

### 3.2 PPO核心公式

#### 3.2.1 Clip Loss（策略损失）

```
L_CLIP(θ) = E[ min(r(θ)·A, clip(r(θ), 1-ε, 1+ε)·A) ]

其中：
  r(θ) = π_θ(a|s) / π_θ_old(a|s)  （概率比率）
  A = Advantage（优势函数）
  ε = 0.1~0.2（clip范围）
```

**min操作的作用**：取未裁剪和已裁剪两者中更保守（悲观）的估计，确保优化目标不会过于激进。

**Clip的作用细节**：
- 当动作被认定为「好」（A > 0）时，算法希望r > 1，但clip将上限限制在1+ε
- 当动作被认定为「坏」（A < 0）时，算法希望r < 1，但clip将下限限制在1-ε
- 防止策略因单次更新过大而导致崩溃（Policy Collapse）

#### 3.2.2 优势函数的计算（GAE）

PPO需要计算每个token的Advantage A_t，使用GAE（Generalized Advantage Estimation）来平衡偏差和方差。

**TD误差**：
```
δ_t = r_t + γ·V(s_{t+1}) - V(s_t)
```
含义：「真实发生的（即时奖励+下一步预测）」减去「这一步原本的预测」

**GAE公式**：
```
Â_t = Σ_{k=0}^∞ (γλ)^k · δ_{t+k}
    = δ_t + (γλ)·δ_{t+1} + (γλ)²·δ_{t+2} + ...
```

其中：
- γ = 折扣因子（通常0.99）
- λ = GAE参数（通常0.95），平衡偏差-方差 trade-off

**直观理解**：
- λ→0：只看一步（TD误差），低方差但偏差大
- λ→1：看无限远（蒙特卡洛），无偏但方差大
- λ=0.95：利用未来信息修正现在，又防止未来噪音过度干扰

**注意**：在大模型RL中，奖励极其稀疏（通常只有最后一个token有奖励），GAE的作用被削弱，很多实现会简化或不用。

#### 3.2.3 价值函数损失（Critic Loss）

用于训练Critic网络，目标是让其对状态价值的预测更准确：

```
L_VF(θ) = (V_θ(s) - V_target)²
```

其中V_target可以通过实际累计回报或TD(λ)等方法估计。

在一些实现中（如DeepSpeed Chat），也会对价值函数更新进行裁剪：

```
L_VF(θ) = 0.5 · E[ max((V-V_target)², (V_clipped-V_target)²) ]

其中 V_clipped = clip(V, V_old-ε, V_old+ε)
```

#### 3.2.4 熵奖励（Entropy Bonus）

```
S[π_θ](s) = -Σ_a π_θ(a|s) log π_θ(a|s)
```

- 熵值越高，策略的动作分布越均匀，探索性越强
- 被加入损失函数（取负号，即最小化负熵 = 最大化熵）
- 鼓励策略保持随机性，防止过早收敛到局部最优

#### 3.2.5 最终总损失函数

```
L_Total(θ) = -L_CLIP(θ) + c₁·L_VF(θ) - c₂·S[π_θ](s) - β·KL[π_ref||π_θ]

其中：
  c₁, c₂ = 超参数，用于平衡三项的重要性
  β = KL惩罚系数（大模型RL中常用，防止偏离SFT模型太远）
```

**三项平衡**：
- **L_CLIP**：让策略跑得好（Actor Loss）
- **L_VF**：让价值网络预测得准（Critic Loss）
- **Entropy**：让策略多尝试新动作（探索）
- **KL惩罚**：防止偏离参考模型太远（大模型RL特有）

### 3.3 PPO训练流程与K步更新机制

#### 3.3.1 完整训练周期

```
Step 1: 采样（Rollout）
    用当前策略π_θ_old与环境交互，收集一批轨迹数据
    记录：states, actions, rewards, logprobs_old（冻结）
    
Step 2: 评估（Compute Returns & Advantages）
    用Critic计算V(s)
    用GAE计算Advantage A_t
    计算Returns用于训练Critic
    
Step 3: 更新（Learning）- K步梯度下降
    for step in range(K):  # K通常=4
        用当前Policy计算logprobs_new
        r = exp(logprobs_new - logprobs_old)  # 重要性采样比
        计算L_CLIP, L_VF, Entropy
        反向传播，更新Policy和Critic参数
        
Step 4: 清空数据，回到Step 1，用新策略重新采样
```

#### 3.3.2 为什么要K步更新？

原始策略梯度是严格的「同策略」方法：采样→更新→数据废弃→重新采样。这导致样本利用效率极低。

PPO通过重要性采样，允许用旧策略的数据训练新策略，但为了效率，会对同一批数据做K步更新：
- 生成回答成本高（需完整前向推理）
- 只更新1步 → GPU利用率低
- K步是**效率与正确性的折中**

#### 3.3.3 ρ的累积漂移

- `logprobs_old`：采样时记录，整个K步内**冻结不变**
- `logprobs_new`：每步用当前参数重新计算
- ρ = π_θ/π_θ_old 衡量「从采样起点到现在的累积偏移」
- ρ单调增大，当ρ > 1+ε，clip截断 → 自动刹车，等下一轮采样

### 3.4 PPO的优缺点

| 优点 | 缺点 |
|------|------|
| 训练稳定，工业界验证最成熟 | 需要4个模型，显存开销大 |
| Critic提供精细的token级指导 | Critic训练难，容易过估计 |
| 适用于各种奖励设置 | 实现复杂，超参数多 |
| 支持在线探索 | 长文本场景效率较低 |

## 第四部分：DPO

### 4.1 核心思想：跳过RL

DPO的洞察：对于语言模型，**奖励函数和策略存在一一对应关系**。既然最终目标是优化策略，为什么不直接从偏好数据优化策略，跳过中间的RM训练？

**传统流程**：偏好数据 → 训练RM → 用PPO优化Policy

**DPO流程**：偏好数据 → 直接优化Policy

### 4.2 DPO核心公式
**损失函数**：
```
L_DPO = -log σ( β · (log(π_θ/π_ref)(y_w) - log(π_θ/π_ref)(y_l)) )
其中：
  y_w = 偏好数据中的好回答
  y_l = 偏好数据中的差回答
  π_ref = 参考模型（冻结的SFT模型）
  β = 温度系数，控制偏离参考模型的程度（通常0.1~0.5）
  σ = sigmoid函数
```
**直观理解**：
- 最大化 `log π_θ(y_w)` → 增加好回答的概率
- 最小化 `log π_θ(y_l)` → 降低差回答的概率  
- `log(π_θ/π_ref)` → 相对于参考模型的改进，防止偏离太远

### 4.3 DPO的核心逻辑

DPO假设：人对回答的偏好可以用**Bradley-Terry模型**描述——好回答的奖励减差回答的奖励越大，人越偏好好回答。

通过数学推导（将奖励用策略表达），可以直接得到Policy的损失函数：

```
对每条偏好数据 (prompt, y_winner, y_loser)：
    最大化 y_winner 的概率
    最小化 y_loser 的概率
    同时不要偏离参考模型太远（KL约束）
```

### 4.4 DPO vs PPO

| 维度 | DPO | PPO |
|------|-----|-----|
| 模型数量 | 2个 (Policy + Reference) | 4个 (+ Critic + RM) |
| 显存占用 | ~50% of PPO | 高 |
| 数据需求 | 偏好对 (y_w, y_l) | 单条+奖励分数 |
| 训练方式 | 离线（直接优化） | 在线（采样+打分） |
| 训练速度 | 快 | 慢（需要rollout） |
| 探索能力 | 较弱 | 较强 |
| 稳定性 | 稳定，不易崩溃 | 需要 careful tuning |

**选择建议**：
- 有现成偏好数据、资源有限 → DPO
- 需要在线探索、奖励信号复杂 → PPO

### 4.5 DPO实践建议

1. **偏好数据质量 > 数量**：高质量1K > 低质量10K
2. **β参数**：控制与参考模型的偏离，通常0.1-0.5
3. **长度归一化**：长文本按token数平均，避免长回答占优
4. **变体方法**：
   - **IPO**：解决DPO过拟合
   - **KTO**：只需二元反馈（好/坏），不需要配对

## 第五部分：GRPO与DAPO

### 5.1 R1复现背景

DeepSeek-R1展示了**不依赖SFT蒸馏，直接用RL激发推理能力**的可能性。核心做法：

- 使用GRPO算法（而非PPO）
- 奖励函数：格式奖励 + 正确性奖励
- 模型自发学会在`<think>`标签内做CoT推理

### 5.2 GRPO：去Critic的策略梯度

#### 5.2.1 为什么去掉Critic？

PPO的Critic在大模型场景下问题多多：
- **显存贵**：Critic和Policy几乎一样大，占50%显存
- **估计难**：文本生成是稀疏奖励，Critic很难准确估计每个token的价值
- **没必要**：同一问题的不同回答可以直接比较，不需要绝对价值估计

#### 5.2.2 GRPO的核心创新

**用组内相对排序代替Critic的绝对估计**：

```
对同一个prompt：
    1. 采样G个回答
    2. 每个回答获得奖励r_i（如0或1）
    3. 计算组内均值μ和标准差σ
    4. Advantage = (r_i - μ) / σ
    5. 用PPO-Clip更新
```

**关键洞察**：
- 不需要知道「这个回答值多少分」
- 只需要知道「这个回答比同组其他回答好多少」
- 组内归一化自动完成了「baseline」的功能

**GRPO的Token级细节**：
虽然Advantage是**序列级**（同一条回答的所有token共享同一个A_i），但：
- **ρ是token级的**：每个token有独立的ρ_{i,t} = π_θ(token_t) / π_θ_old(token_t)
- **clip是token级的**：每个token独立判断是否超出[1-ε, 1+ε]
- Loss对所有token的min(ρ·A, clip(ρ)·A)求和

**为什么这样设计？**
- 同一条回答的所有token「共享荣辱」（都好或都差）→ 序列级A足够
- 但每个token的概率变化不同 → 需要token级ρ来精细调节
- 这是GRPO与PPO的关键区别：PPO是token级A + token级ρ，GRPO是序列级A + token级ρ

#### 5.2.3 GRPO vs PPO

| 特性 | PPO | GRPO |
|------|-----|------|
| 模型数量 | 4个 | 2个 |
| Critic | 需要 | **去掉** |
| Advantage | Critic估计V，用GAE计算 | 组内归一化 |
| 显存 | 高 | **省50%+** |
| 实现 | 复杂 | **简单** |
| 适用场景 | 通用 | 可验证任务（数学/代码）|

#### 5.2.4 Group Size的选择

| Group Size | 效果 |
|------------|------|
| 2-4 | 太小，梯度方差大，不稳定 |
| 8-16 | 推荐，平衡成本和稳定性 |
| 32-64 | 最佳，R1论文使用 |
| 128+ | 边际收益递减 |

**类比**：看2个人的考试成绩排名没意义，看64个人的排名才有统计意义。

### 5.3 DAPO：GRPO的工程优化

GRPO简单高效，但训练不稳定（Entropy Collapse、无效梯度等）。DAPO是字节+清华2025年提出的改进，在GRPO基础上加入4个trick。

#### 5.3.1 四项核心改进

**1. Clip-Higher（非对称裁剪）**

```
PPO/GRPO: clip(r, 1-0.2, 1+0.2)  上下界对称
DAPO:     clip(r, 1-0.2, 1+0.28) 上界更宽松
```

- 上界放宽（0.28）→ 鼓励探索，防止过早收敛
- 下界保持（0.2）→ 抑制坏行为，保持稳定

**2. Dynamic Sampling（动态采样过滤）**

过滤掉「全对」或「全错」的组：
- 组内全对 → 所有A=0 → 梯度为0 → 浪费计算
- 组内全错 → 同理

DAPO在训练过程中**实时检测并过滤**这类无效样本，同时从数据池中动态补充新的prompt进来，保证每个batch都有足够的有效梯度信号，训练效率提升2-3倍，相同效果所需训练步数减半。

**3. Token-Level Loss**

```
GRPO: 按样本平均 → 长回答吃亏
DAPO: 按token平均 → 长短公平
```
明确将损失归一化到所有有效 token 的总数，而非先按序列长度归一再跨序列平均，避免短序列和长序列对梯度贡献相等

**4. Overlong Reward Shaping**

当模型生成的回答超过max_length被强制截断时，我们无法判断答案对错，这个奖励是噪声。GRPO直接给0或给格式分，会误导模型。

DAPO对截断的回答施加**连续性惩罚**（如-0.5），而不是简单给0。这明确告诉模型「生成太长被截断是坏事」，促使模型学会在长度限制内完成推理。

**效果**：减少reward noise，训练曲线更平滑，少出现「快答对了却被截断」的遗憾情况。

**5. Remove KL**

DAPO 认为在训练长思维链推理模型时，模型分布可能与初始模型显著偏离，因此这种限制不是必要的。

#### 5.3.2 DAPO性能

| 指标 | R1-Zero | DAPO |
|------|---------|------|
| AIME 2024 | 47分 | **50分** |
| 训练步数 | - | **减半** |
| 模型 | DeepSeek-R1 | Qwen2.5-32B |

### 5.4 Dr.GRPO：修正GRPO的优势估计偏差

**背景**：Dr.GRPO是与DAPO同期的工作，直接面向GRPO的原始问题进行改进（而非DAPO）。

#### 5.4.1 动机：GRPO的两个偏差问题

**1. 响应长度偏差（Response-level length bias）**

GRPO目标函数中的 $1/|o_i|$ 项（按序列长度归一化）引入了系统性偏差：

- **当优势 $\hat{A}_{i,t} > 0$（回答正确）时**：较短回答因 $1/|o_i|$ 更大而产生更大的梯度更新，策略会**偏好更短的正确回答**
- **当优势 $\hat{A}_{i,t} < 0$（回答错误）时**：较长回答因 $1/|o_i|$ 更小而受到更小的惩罚，策略会**在错误回答中偏好较长的回答**

这种偏差导致模型对回答长度的偏好与正确性耦合，不是最优策略。

**2. 问题难度偏差（Question-level difficulty bias）**

GRPO优势函数的分母是组内奖励的标准差 $\text{std}(R(q, o_1), ..., R(q, o_G))$：

- 标准差较低的问题（特别简单或特别困难、奖励几乎总是1或0）会得到**更大的权重**
- 与常见的advantage normalization（按batch归一化）不同，这种按问题归一化会让不同问题的样本在目标函数中具有差异化的权重，导致**难度相关偏置**

#### 5.4.2 实现：移除两个偏差项

**修正方案**：
1. **移除长度归一化**：在计算策略梯度时不除以回答长度 $|o_i|$
2. **移除标准差归一化**：在计算优势时不除以奖励的标准差

**修正后的优势估计**：
```
Â_{i,t} = R(q, o_i) - mean({R(q, o_1), ..., R(q, o_G)})
```

**修正后的GRPO目标函数**：
```
J_Dr.GRPO(θ) = (1/G) Σ_{i=1}^G Σ_{t=1}^{|o_i|} min[ (π_θ/π_old)·Â, clip(π_θ/π_old, 1-ε, 1+ε)·Â ]
```

#### 5.4.3 后续改进：GPG

GPG（Generalized Policy Gradient）进一步改进了Dr.GRPO的优势缩放问题：

```
A_GPG = α * (R(o) - mean{R(o)})
```

其中 $α$ 是可学习的缩放系数，解决了Dr.GRPO直接去掉标准差可能过于草率的问题。

| 方法 | 优势函数 | 特点 |
|------|----------|------|
| PPO | GAE-based | 依赖Critic模型 |
| GRPO | $(R - \mu) / \sigma$ | 组内归一化，但有偏差 |
| Dr.GRPO | $R - \mu$ | 移除长度和标准差偏差 |
| DAPO | $(R - \mu) / \sigma$ | 保留GRPO，从其他方面优化 |
| GPG | $\alpha * (R - \mu)$ | 可学习缩放因子 |

### 5.5 GFPO：多采样少思考

**动机**：RLVR（Reinforcement Learning with Verifiable Rewards）模式训练的模型为了获得"正确奖励"，往往生成冗长答案，但大部分长答案不能提升正确性。

#### 5.5.1 RLVR推理模型的问题

现有RL+RLVR训练存在以下问题：
- **冗长推理痕迹**：模型为获得正确奖励，生成极其冗长的reasoning trace
- **无意义填充**：很多token是重复的、无意义的filler
- **长度与正确性解耦**：虽然长答案能提高准确率，但大多数额外长度与正确性无关甚至有害

**实验证据**：
- 在72%的AIME25问题中，较长回答更容易是错误的
- GRPO训练后的模型长度从4k暴涨到14k tokens

#### 5.5.2 GFPO实现：拒绝采样+简洁性子集

GFPO本质上是一种**Rejection Sampling（拒绝采样）**算法，在训练时多采样，从中筛选"简洁性"子集，只在该子集内使用GRPO：

**算法流程**：
1. **评分**：对每个回答 $o_i$ 计算简洁性评分：$\text{scores} = [\text{metric}(o_i)]_{i=1}^G$
2. **排序**：根据给定排序方向对scores排序，得到排序后的索引：$\text{idx} = \text{ARGSORT}(\text{scores}, \text{order})$
3. **构造保留集合**：从排序后的前k个索引中构造保留集合：$\mathcal{S} = [\text{idx}[j]]_{j=1}^k$
4. **构造掩码**：为整个组构造0/1掩码，属于 $\mathcal{S}$ 的位置为1，否则为0：$m = [\mathbb{I}(i \in \mathcal{S})]_{i=1}^G$
5. **计算优势**：只在掩码 $m$ 下计算优势

**形式化表达**：
```
S = REJECTIONSAMPLE(G, k, metric),  m_i = I_{i∈S}

Â_{i,t}^{(m)} = (R(q,o_i) - mean{R(q,o_s1),...,R(q,o_sk)}) / std{R(q,o_s1),...,R(q,o_sk)} * m_i

L_GFPO = min[(π_θ/π_old)·Â, clip(π_θ/π_old, 1-ε, 1+ε)·Â]
```

**关键思想**：通过筛选简洁的子集进行训练，鼓励模型生成简洁且正确的回答，而非冗长的"思考"。

### 5.6 GSPO：面向MoE架构的重要性采样对齐

**动机**：之前的GRPO及其改进方法存在**重要性采样和奖励颗粒度不对齐**的问题，在MoE（Mixture of Experts）结构中尤为严重。

#### 5.6.1 不对齐问题

**1. Per-Token vs Sequence粒度不对齐**
- **重要性采样**：GRPO的重要性采样是Per-Token粒度的（$\frac{\pi_\theta(o_{i,t}|q,o_{i,<t})}{\pi_{\theta_{\text{old}}}(o_{i,t}|q,o_{i,<t})}$）
- **Reward**：但Reward却是Sequence粒度的（整个回答只有一个分数）
- **结果**：两者不对齐，重要性采样的有效性依赖于多次采样，但在token级别的单次采样会使重要性权重产生噪声

**2. MoE结构中的问题**
- 在MoE结构中，新旧策略可能会激活不同的Expert
- 这进一步增大Per-Token采样带来的噪声

#### 5.6.2 实现：Sequence-Level重要性采样

GSPO采用**Sequence-Level的重要性采样**来对齐Reward粒度：

**重新定义的重要性比率**：
```
s_i(θ) = (π_θ(y_i|x) / π_θ_old(y_i|x))^(1/|y_i|)
       = exp( (1/|y_i|) Σ_{t=1}^{|y_i|} log(π_θ(y_{i,t}|x,y_{i,<t}) / π_θ_old(y_{i,t}|x,y_{i,<t})) )
```

这是Per-Token重要性比率的几何平均，实现了长度归一化以减小偏差。

**GSPO目标函数**：
```
J_GSPO(θ) = E[ (1/G) Σ_{i=1}^G min( s_i(θ)·Â_i, clip(s_i(θ), 1-ε, 1+ε)·Â_i ) ]
```

**优势估计**（与GRPO相同，使用组内归一化）：
```
Â_i = (r(x,y_i) - mean({r(x,y_i)}_{i=1}^G)) / std({r(x,y_i)}_{i=1}^G)
```

**与GRPO的关键区别**：
| 特性 | GRPO | GSPO |
|------|------|------|
| 重要性采样 | Per-Token | Sequence-Level |
| 粒度 | Token | Sequence |
| 适用架构 | Dense | MoE |
| 长度归一化 | 在Loss中 | 在重要性比率中 |

### 5.7 R1复现实践经验

#### 奖励函数设计

```
总奖励 = 格式奖励 + 正确性奖励

格式奖励：
    - 包含<think>...</think><answer>...</answer>结构 → +0.1~0.5
    
正确性奖励：
    - 答案正确 → +1
    - 答案错误 → 0
```

**关键**：两者缺一不可。只有正确性奖励，模型不会自发产生CoT；只有格式奖励，推理内容可能是垃圾。

#### 常见问题

| 问题 | 现象 | 解决 |
|------|------|------|
| **中英混答** | 回答中英文混杂 | 语言一致性奖励；增加中文样本 |
| **Reward Hacking** | 冗长无意义的思考骗格式分 | 简化格式奖励；长度惩罚 |
| **Entropy Collapse** | 输出多样性下降，趋同 | DAPO的Clip-Higher；提高温度 |
| **训练不稳定** | Loss震荡 | 增大Group size；动态采样 |

#### 训练框架

| 框架 | 特点 |
|------|------|
| **verl** | 字节/清华开源，DAPO基础框架，适合大规模 |
| **OpenRLHF** | 开源RLHF/GRPO框架 |
| **TRL** | HuggingFace出品，轻量易用 |

## 第六部分：面试高频问题速查

### 通用问题

**Q: SFT和RL的本质区别？**
> SFT是模仿学习，学「老师怎么做」；RL是试错学习，学「怎样获得高奖励」。SFT权重固定，RL用奖励加权。

**Q: 为什么RL需要参考模型？**
> 防止策略偏离原始能力太远（KL约束）。没有参考模型，可能为了迎合奖励而遗忘基础能力。

### PPO

**Q: PPO的4个模型各是什么作用？**
> Policy：被训练的策略；Reference：提供KL约束基准；Critic：估计token级价值；Reward Model：给回答打分。

**Q: PPO的Clip有什么用？**
> 限制单次更新幅度，防止策略崩溃。当概率比率超出[1-ε,1+ε]时梯度被截断。

**Q: PPO为什么要对同一批数据做K步更新？**
> 采样成本高（生成回答需完整推理）。只更新1步GPU利用率太低，K步（通常4）是效率与正确性的折中。ρ的clip机制自动防止数据过期。

**Q: ρ（重要性采样比）的累积漂移是什么意思？**
> logprobs_old在整个K步内冻结不变，ρ衡量的是「从采样起点到现在的累积偏移」，不是单步偏移。当累积偏移超过1+ε，clip自动截断，等下一轮采样。

**Q: 大模型PPO和传统PPO有什么不同？**
> 大模型奖励极其稀疏（只有最后有分），GAE作用被削弱；Critic估计不准；主要瓶颈是显存（4个模型）。

### DPO

**Q: DPO为什么不需要显式奖励模型？**
> DPO从偏好数据直接推导Policy的损失函数，跳过了RM训练。核心洞察：奖励和策略存在对应关系。

**Q: DPO适合什么场景？**
> 有现成偏好数据、计算资源有限、需要快速迭代的场景。不适合需要强在线探索的场景。

### GRPO与DAPO

**Q: GRPO和PPO最大的区别？**
> GRPO去掉了Critic网络，用组内多个采样的相对排序估计Advantage。省50%显存，更适合稀疏奖励场景。

**Q: PPO的Advantage和ρ是token级还是序列级？GRPO呢？**
> - PPO：都是token级。GAE给每个token计算A_t，每个token有独立的ρ_t和clip
> - GRPO：Advantage是序列级（同一条回答共享A_i），但ρ和clip仍是token级（每个token独立计算）
> - 关键区别：GRPO用序列级A简化了优势估计，但保留了token级ρ来精细调节每个token的更新

**Q: GRPO的Advantage怎么计算？**
> 同一prompt采样G个回答，A_i = (r_i - 组内均值) / 组内标准差。本质是组内相对排序。

**Q: Group size过小会怎样？**
> 梯度方差大，Advantage估计噪声大，训练不稳定，易reward hacking。建议至少8-16。

**Q: DAPO的四项改进是什么？**
> 1. Clip-Higher：非对称裁剪，鼓励探索；2. Dynamic Sampling：过滤无效样本；3. Token-Level Loss：长短回答公平对待；4. Overlong Shaping：惩罚超长截断。

**Q: Dr.GRPO解决了GRPO的什么问题？**
> Dr.GRPO修正了GRPO的两个偏差：1. 响应长度偏差（按长度归一化导致短回答梯度大）；2. 问题难度偏差（按问题标准差归一化导致简单/困难问题权重过大）。

**Q: GFPO的核心思想是什么？**
> GFPO通过拒绝采样筛选"简洁性"子集，只在该子集内使用GRPO训练，鼓励模型生成简洁且正确的回答，解决RLVR训练中答案冗长的问题。

**Q: GSPO与GRPO的主要区别？**
> GSPO采用Sequence-Level的重要性采样（几何平均）而非Per-Token采样，解决了重要性采样与Sequence-Level奖励的粒度不对齐问题，特别适合MoE架构。

**Q: 为什么DAPO能超越R1-Zero？**
> 通过四项工程优化解决了GRPO的训练不稳定问题，在Qwen2.5-32B上达到50分（vs 47分），训练步数减半。

### R1复现

**Q: R1复现的奖励函数怎么设计？**
> 格式奖励（鼓励CoT结构）+ 正确性奖励（答案对错）。两者缺一不可。

**Q: 数据泄露怎么考虑？**
> n-gram去重、embedding相似度检测、多样化题目变体、held-out验证。

**Q: 中英混答怎么解决？**
> 语言一致性奖励、格式奖励中惩罚混杂、增加中文样本。

## 第七部分：算法选择决策树

```
开始: 你的场景是?
│
├─> 有偏好对数据 (y_w, y_l)
│   └─> 资源有限，追求稳定
│       └─> DPO / KTO
│
├─> 有奖励信号，可验证任务（数学/代码）
│   ├─> 追求极致效率，省显存
│   │   └─> GRPO
│   ├─> 追求训练稳定
│   │   └─> DAPO (推荐)
│   ├─> 需要更准确的优势估计
│   │   └─> Dr.GRPO
│   ├─> 答案过于冗长，需要简洁性
│   │   └─> GFPO
│   ├─> MoE架构模型
│   │   └─> GSPO
│   └─> 有充足资源，追求精度（token级别）
│       └─> PPO
│
├─> 复杂奖励，在线探索
│   └─> PPO
│
└─> 无明确奖励，只有人类反馈
    └─> 先训练Reward Model → 再选PPO/GRPO/DAPO
```

**快速选择指南**：
- **快速迭代/资源有限** → DPO
- **数学/代码等可验证任务** → GRPO / DAPO
- **复杂奖励/强探索需求** → PPO
- **GRPO训练不稳定** → 换DAPO

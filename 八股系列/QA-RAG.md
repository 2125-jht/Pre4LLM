# RAG

---

## 1. 请解释 RAG 的工作原理。与直接对 LLM 进行微调相比，RAG 主要解决了什么问题？有哪些优势？

**RAG (Retrieval-Augmented Generation)** 是一种"**先检索，后生成**"的模式：

1. **检索**：将用户问题作为查询，在外部知识库中搜索最相关的文档片段
2. **增强**：将检索到的信息与用户问题拼接，形成增强提示
3. **生成**：LLM 基于上下文生成准确、有事实依据的回答

**解决的核心问题：**

| 问题 | 说明 |
|------|------|
| **知识静态性** | LLM 知识被"冻结"在训练截止点，RAG 通过可更新知识库获取最新信息 |
| **幻觉** | LLM 倾向于捏造事实，RAG 将回答"锚定"在具体上下文上 |
| **缺乏专业/私有知识** | 微调成本高昂，RAG 可轻松连接私有数据集 |

**与微调相比的优势：**
- **知识更新成本低**：只需更新文档，无需重新训练模型
- **可追溯性**：可展示答案来源，便于事实核查
- **降低幻觉**：提供事实依据
- **高性价比**：注入事实性知识成本远低于微调
- **个性化**：可为不同用户动态接入不同知识源

## 2. 一个完整的 RAG 流水线包含哪些关键步骤？

**阶段一：数据准备 / 索引流水线（离线）**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1. 数据加载 | Load | 从 PDF、Word、网页等加载原始文档 |
| 2. 文本切块 | Split / Chunk | 将长文档切割成语义完整的小块 |
| 3. 嵌入 | Embed | 使用嵌入模型将文本块转为高维向量 |
| 4. 存储 | Store | 存入向量数据库（FAISS、ChromaDB、Pinecone 等） |

**阶段二：查询 / 推理流水线（在线）**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1. 用户提问 | User Query | 接收用户输入的自然语言问题 |
| 2. 查询嵌入 | Embed Query | 使用**相同**的嵌入模型将问题转为查询向量 |
| 3. 向量检索 | Retrieve | 计算相似度，找出 Top-K 个最相关的文本块 |
| 4. 重排序（可选）| Re-rank | 使用 Cross-Encoder 精细排序，选出 Top-N（N < K）|
| 5. 增强与生成 | Augment & Generate | 将最优文本块与问题组合成提示，交由 LLM 生成答案 |

## 3. 在构建知识库时，如何选择合适的切块大小和重叠长度？

**选择切块大小：**
- **依据嵌入模型能力**：切块应小于模型最大 Token 限制，中等长度（256-512 token）通常效果最佳
- **依据数据类型**：
  - 结构化文档（论文、报告）：采用语义切块（按段落、标题）
  - 非结构化长文本：依赖固定长度切块
  - 代码：按函数或类切块
- **依据查询类型**：具体问题适合小块，宽泛问题适合大块

**选择重叠长度：**
- **作用**：防止语义信息在切块边界被切断
- **经验法则**：设置为切块大小的 **10%-20%**

**大块 vs 小块权衡：**

| 维度 | 大块 | 小块 |
|------|------|------|
| **优点** | 包含更丰富上下文 | 信息密度高，检索精确 |
| **缺点** | 噪声增加、成本高、"Lost in the Middle"问题 | 上下文不足，语义易割裂 |

> **实践建议**：从基线（`chunk_size=512`, `overlap=64`）开始，针对具体场景迭代优化。

## 4. 如何选择一个合适的嵌入模型？评估 Embedding 模型的好坏有哪些指标？

**选择方法：**

1. **参考排行榜**
   - **MTEB**：最权威的嵌入模型评测基准，关注**检索**任务得分
   - **C-MTEB**：专门针对中文
2. **考虑应用场景**
   - 专业领域优先考虑领域预训练模型
   - 确保支持业务涉及的语言
   - 在效果和性能间权衡
3. **私有 vs 开源**
   - **私有模型（如 OpenAI Ada）**：性能好但存在隐私风险和成本
   - **开源模型（如 BGE、M3E）**：可本地部署、数据安全

**评估指标：**

| 任务 | 核心指标 | 说明 |
|------|---------|------|
| **检索** | nDCG@k | 综合衡量相关性和排名 |
| | Recall@k | 前 k 个结果召回的相关文档比例 |
| | MRR | 第一个相关文档排名的倒数平均值 |
| **语义相似度** | Spearman/Pearson | 向量相似度与人类判断的相关性 |
| **分类** | Accuracy | 嵌入向量作为特征的分类表现 |

## 5. 除了基础的向量检索，还有哪些可以提升 RAG 检索质量的技术？

**一、增强检索器**

**1. 混合搜索（Hybrid Search）**
- 稀疏检索（BM25）+ 密集检索（向量搜索）
- 兼顾关键词精确匹配和语义模糊匹配

**2. 重排序（Re-ranking）**
- 两阶段流程：
  1. **召回**：用快速方法召回较大候选集（Top 50）
  2. **重排**：用 Cross-Encoder 精细排序，选出最终 Top-N
- Cross-Encoder 直接比较查询和文档，捕捉细粒度相关性

**二、优化查询**

**查询扩展与转换**
- **多查询检索**：让 LLM 生成多个角度查询，合并结果
- **HyDE**：让 LLM 生成假设答案，再用答案嵌入去检索
- **子问题查询**：将复杂问题分解为多个简单子问题

**三、优化索引结构**

**1. 小块引用大块（Small-to-Large）**
- 索引时切成小"摘要块"，但保留对"父块"的引用
- 匹配小块获得高精度，送 LLM 的是包含更丰富上下文的父块

**2. 图索引**
- 用 LLM 提取实体和关系，构建知识图谱
- 先在图谱中进行结构化查询，再结合向量检索

## 6. 请解释"Lost in the Middle"问题。它描述了 RAG 中的什么现象？有什么方法可以缓解？

**现象描述：**

LLM 处理长上下文时，倾向于**更好地回忆开头和结尾的信息，而忽略中间部分**。

**在 RAG 中的影响：** 即使检索到相关文档，若位于中间位置，也很可能被 LLM 忽略，导致重排序效果大打折扣。

**缓解方法：**

| 方法 | 核心思想 |
|------|---------|
| **文档重排序** | 将最相关文档放置在上下文**开头和结尾**，次要文档放中间 |
| **减少检索数量** | 严格控制 Top-K 的 K 值（如 Top-3/5），确保召回质量高 |
| **指令化提示** | 在 prompt 中加入："请确保回答完全基于以上所有上下文" |
| **微调 LLM** | 构建特定数据集训练 LLM 利用中间内容（成本最高但最根本） |

## 7. 如何全面地评估一个 RAG 系统的性能？

**检索阶段评估：**

| 指标 | 说明 |
|------|------|
| **上下文精确率** | 检索文档中真正相关的比例 |
| **上下文召回率** | 所有相关文档中被成功找回的比例 |
| **Hit Rate** | 检索结果是否至少包含一个相关文档 |
| **MRR** | 第一个相关文档排名的倒数平均值 |
| **nDCG@k** | 最全面的指标，考虑相关性等级和排名 |

**生成阶段评估：**

| 指标 | 评估问题 |
|------|---------|
| **忠实度** | 答案是否完全基于上下文？是否存在幻觉？ |
| **答案相关性** | 答案是否直接回答了用户问题？ |
| **答案正确性** | 答案信息是否事实准确？ |

**自动化评估框架：**
- **RAGAS**、**ARES**、**TruLens**：使用 LLM-as-a-Judge 自动化计算指标

## 8. 在什么场景下，你会选择使用图数据库或知识图谱来增强或替代传统的向量数据库检索？

向量数据库擅长**语义相似度**匹配，知识图谱擅长**实体与关系**精确查询。

| 场景 | 说明 | 示例 |
|------|------|------|
| **多跳推理** | 问题需要沿关系链多次"跳转" | "Llama 2 作者所在公司的 CEO 是谁？" |
| **强结构关联数据** | 数据包含大量实体和明确关系 | 金融股权结构、医疗药物-基因-疾病关系 |
| **高度可解释性** | 需要清晰解释答案如何得出 | 风控："为什么标记为高风险？" |

**与向量数据库的关系：**

**互补增强**：先用知识图谱找到核心实体，再用结构化信息构建更精确的向量查询，最后汇总给 LLM。

## 9. 传统 RAG 流程是"先检索后生成"，你是否了解一些更复杂的 RAG 范式？

**1. 迭代式检索（Self-RAG、Corrective-RAG）**
- 首次检索生成初步答案
- LLM 反思评估信息是否足够
- 若不足，**主动生成新查询**进行二次检索
- 整合新旧信息生成最终答案

**2. 自适应检索（FLARE、Self-Ask）**
- 不在生成前一次性检索，而是**"按需"即时检索**
- LLM 生成时预测到不确定信息即暂停
- 主动提问并检索，填入结果继续生成
- **优势**：只在需要时检索，避免预先检索大量无关信息

**3. 多源数据 RAG**
- Agent 智能地从多种数据源检索整合：
  - 向量数据库（非结构化文档）
  - 知识图谱（结构化实体关系）
  - SQL 数据库（精确统计数据）
  - 搜索引擎 API（实时信息）
- **本质**：Agent 驱动的 RAG

## 10. RAG 系统在实际部署中可能面临哪些挑战？

| 挑战类别 | 具体问题 |
|---------|---------|
| **数据处理与维护** | 分块策略泛化性；知识库实时更新与向量索引同步 |
| **性能瓶颈** | 延迟："检索+生成"天然比直接调用慢；成本：嵌入、向量库、LLM API 的持续支出 |
| **评估与监控** | 生产环境难获标准答案；需监控性能衰退并定位问题模块 |
| **无答案处理** | 知识库无答案时，系统易强行生成误导性答案；需具备判断相关性并拒绝回答的能力 |
| **安全与隐私** | 访问控制（不同用户对不同文档权限）；提示注入攻击防护 |

## 11. 了解搜索系统吗？和 RAG 有什么区别？

| 特征 | 搜索系统 | RAG 系统 |
|------|---------|---------|
| **任务** | 找文档 | 给答案 |
| **输出** | 文档列表 | 自然语言答案 |
| **用户角色** | 主动，需自己阅读总结 | 被动，直接获得成品 |
| **核心组件** | 索引器 + 排序器 | [索引器 + 排序器] + **生成器(LLM)** |

**关系：**

RAG 是构建在搜索系统之上的更高级应用，**包含**搜索系统作为其"检索"模块。

**比喻：**
- 搜索系统 = 图书馆管理员：告诉你哪些书可能有用，自己去看
- RAG = 历史学专家：查阅书籍后直接告诉你答案

## 12. 知道或者使用过哪些开源 RAG 框架？如何选择合适场景？

**框架类型：**

| 类型 | 代表 | 特点 |
|------|------|------|
| **代码库框架** | LangChain、LlamaIndex | 高度灵活可定制 |
| **端到端平台** | RAGFlow、Haystack、DSPy | 开箱即用，对业务人员友好 |

**RAGFlow 特点：**
- 自动化与可视化：智能文本分块、GUI 界面
- 端到端整合：从数据接入到应用接口的一体化方案

**选择策略：**

| 场景 | 推荐框架 | 理由 |
|------|---------|------|
| 高度定制化 | LangChain / LlamaIndex | **灵活性**和**控制力** |
| 快速原型验证 | RAGFlow / Haystack | **效率**和**易用性** |

**实践建议：**

项目初期用 RAGFlow 等快速搭建**基线**验证业务价值；验证后如需深度优化，再用 LangChain/LlamaIndex 进行精细化**重构和实现**。
